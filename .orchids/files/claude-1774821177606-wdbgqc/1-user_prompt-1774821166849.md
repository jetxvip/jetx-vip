JetX.vp — Final Production Readiness Verification (Do Not Change Working Features)

We have now finished fixing the bugs from the previous deployment, and before doing the next/final deployment we need one last full production-readiness verification.

This is NOT a request for redesign or new feature work.

This is a strict request to verify that the system is stable, safe, and deployment-ready, without breaking anything that already works.

---

## Core instruction

Do NOT introduce unnecessary changes.

Only verify, harden, and fix anything that is still required for a stable final production deployment.

Everything that currently works must continue to work exactly as it does now.

---

## 1. Full production-readiness audit

Please do a final verification of the entire project and confirm that it is ready for production deployment.

This verification must cover:

* public site pages
* admin panel
* authentication
* lead system
* subscribers
* client flow
* flights / offers / content sync
* mobile behavior
* incognito / fresh session behavior
* cross-device sync
* Vercel compatibility
* KV storage compatibility

---

## 2. Deployment safety

Verify that the final codebase is safe for deployment and contains no development-only dependencies or temporary logic that could break production.

Make sure:

* no `.env.local` dependency is required in production
* no dev server files are required
* no debug-only code is required
* no local-only assumptions remain
* no hidden dependency on localStorage exists for critical production behavior

Production must work from Vercel environment variables and KV only.

---

## 3. Authentication / admin verification

Verify all of the following:

* `/login` works correctly
* `/admin` is protected
* no admin access without valid auth
* JWT flow works correctly in production
* session timeout works correctly
* logout works correctly
* admin reads/writes protected data correctly
* public site cannot access private admin-only data

---

## 4. Public site data verification

Verify that all public data is loaded correctly from the production data source and not from local fallback only.

This includes:

* company/contact data
* flights
* Empty Leg flights
* Special Offers
* fleet
* dynamic content

Fresh mobile and incognito sessions must show the same current data as the admin-managed site.

---

## 5. Lead / subscriber verification

Verify that the following all work correctly in production:

* quote requests
* Empty Leg inquiries
* Special Offer inquiries
* contact form submissions
* Empty Leg subscription form

All of them must:

* be stored correctly
* reach admin correctly
* appear in a readable format
* work from desktop, mobile, and incognito
* not require refresh in admin to become visible
* trigger notifications correctly

---

## 6. Admin notification verification

Verify that when admin is open and a new lead arrives from another device:

* the lead appears automatically
* bell/unread indicator updates
* notification sound plays
* no manual refresh is required

This must work reliably in the final deployment.

---

## 7. Client workflow verification

Verify the lead-to-client workflow carefully.

This includes:

* client creation
* closing workflow
* warning when trying to close a non-client lead
* “Create Client Now” flow if implemented
* client card creation
* revenue/flight history tracking
* editing and deleting client flight history entries

This flow must remain clean and consistent.

---

## 8. Defensive stability

Verify that no route can white-screen or crash due to:

* undefined data
* malformed KV values
* missing translations
* empty arrays
* missing props
* route-specific runtime errors

Important routes to verify:

* home
* fleet
* empty legs
* special offers
* ambulance flights
* contact
* login
* admin

The site must fail gracefully, never collapse into a white screen.

---

## 9. Cross-device production verification

Please verify behavior in a realistic production scenario:

1. admin open on desktop
2. user submits data from mobile/incognito
3. admin receives it automatically
4. public site on another device reflects updates correctly
5. all changes persist after refresh

This must be validated as a real production flow.

---

## 10. KV / storage verification

Verify that:

* KV is the real source of truth in production
* public data loads correctly from KV
* no double-serialization or malformed data issue remains
* arrays/objects are normalized correctly on read
* old malformed data cannot crash the app
* private keys remain protected
* public keys remain readable as intended

---

## 11. Final deployment verification

Before declaring the system ready, verify:

* clean build
* no breaking console/runtime errors
* no broken routes
* no missing environment-variable assumptions
* no production blockers remaining

---

## Final instruction

Only confirm “production-ready” if the system is truly safe for final deployment and the full flow works without hidden breakage.

If anything is still risky, fragile, or incomplete, identify it explicitly before deployment.
