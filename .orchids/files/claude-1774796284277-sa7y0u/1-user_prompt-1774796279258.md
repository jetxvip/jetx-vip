Root cause confirmed — public API returns stringified JSON inside `value`

I tested the public production endpoints directly:

* `/api/store?key=jetx_company`
* `/api/store?key=jetx_offers`
* `/api/store?key=jetx_flights`

They do return data successfully, but the payload is still wrapped as a JSON string inside `value`.

Examples:

* company returns:
  `"value":"{\"phone\":\"...\"}"`
* flights/offers return:
  `"value":"[{...}]"`

This means the frontend must parse `data.value`, otherwise fresh sessions (mobile/incognito) do not get usable objects/arrays and fall back to default/empty state.

## Required fix

### Option A (preferred)

Fix `/api/store` GET so it returns parsed JSON, not stringified JSON inside `value`.

Expected shape:

* company:
  `"value": { ... }`
* flights/offers/fleet:
  `"value": [ ... ]`

### Option B

If keeping the current API shape, then frontend must always do:

```js
const raw = data.value;
const parsed = typeof raw === 'string' ? JSON.parse(raw) : raw;
```

And then validate:

* arrays with `Array.isArray(parsed)`
* objects with safe object checks

## Important

This is the reason why:

* admin browser appears correct
* but fresh mobile/incognito sessions show base/default content

Because local state/localStorage hides the issue on the admin/browser that already has data.
