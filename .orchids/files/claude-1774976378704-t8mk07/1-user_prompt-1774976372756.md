ORCHID IMPLEMENTATION PROMPT

Build a mandatory legal consent modal that appears before EVERY form submission on the website.

This must be implemented globally across the site and must work for all forms, including:

* contact forms
* lead forms
* quote request forms
* booking / inquiry forms
* any future form that sends data to us

CORE RULE:
The user must approve the legal consent checkbox before every single submission.
Do not reuse a previous approval.
Do not persist approval across submissions.
Do not skip the modal if the user already approved before.
Do not use sessionStorage, localStorage, cookies, memory cache, or any other mechanism to bypass repeated approval.
This approval must happen again every time the user submits any form.

LANGUAGE LOGIC:
The website has 2 language modes only:

* Hebrew
* English

The legal modal must always match the currently active site language at the moment the user clicks submit.

If current site language is Hebrew:

* modal content language = Hebrew
* checkbox text = Hebrew
* button labels = Hebrew
* legal text = Hebrew
* layout direction = RTL
* text alignment = right

If current site language is English:

* modal content language = English
* checkbox text = English
* button labels = English
* legal text = English
* layout direction = LTR
* text alignment = left

Do not mix languages inside the same modal.

SUBMISSION FLOW:

1. User fills any form.
2. User clicks the form submit button.
3. Prevent immediate form submission.
4. Open a modal dialog.
5. Inside the modal, display the full legal text in a scrollable content box.
6. Under the legal text, display one mandatory checkbox.
7. Keep the primary action button disabled until the checkbox is checked.
8. Once user checks the checkbox and clicks confirm, continue with the original form submission.
9. If user closes modal or clicks back/cancel, do not submit the form.

MUST APPLY TO:

* mouse click submit
* tap submit on mobile
* Enter key submission
* any programmatic submit triggered by UI actions

Prevent bypassing the modal in all normal frontend flows.

UI / UX REQUIREMENTS:
The modal must feel premium, minimal, neutral, and not aggressive.

Design requirements:

* centered modal
* white background
* soft shadow
* rounded corners
* subtle border if needed
* no warning icons
* no red theme
* no alarming legal tone in the visual design
* no oversized bold legal headings
* elegant spacing
* polished luxury-style UI
* responsive on desktop and mobile

The goal is:

* legally strong
* visually calm
* not intimidating
* easy to approve
* not sending user to another page

MODAL CONTENT STRUCTURE:
Inside the modal, render in this exact order:

1. Optional minimal title:
   If Hebrew:
   "אישור תנאים לפני שליחה"
   If English:
   "Confirmation Before Submission"

2. Scrollable legal content box containing:

* full Israel legal text
* immediately followed by full Europe / EU / GDPR legal text
* one after the other inside the same scrollable container
* no external links
* no separate page
* no new tab
* no “read more” redirect
* the full text must appear directly inside the modal

3. Mandatory checkbox under the scroll box

4. Action buttons row under the checkbox

If English mode, use EXACTLY:
"I confirm that I have read and agreed to the Terms of Use, Privacy Policy, and Limitation of Liability of the website."

Do not change this wording.

Do not add explanatory text around it.

* clicking primary button submits the original form

TECHNICAL IMPLEMENTATION REQUIREMENTS:
Use a reusable global component, for example:

* LegalConsentModal
* useLegalConsentInterceptor
* submitWithLegalConsent(formElement)

Recommended architecture:

1. Intercept all form submissions centrally
2. Capture the original form reference and submit intent
3. Open modal
4. Wait for explicit approval
5. Resume submission using the original form context

Implementation should be robust enough for:

* plain HTML forms
* React controlled forms
* async submit handlers
* forms with validation
* forms inside modals
* multiple forms on the same page

VALIDATION ORDER:
Recommended order:

1. form client-side validation runs first
2. if form is valid and ready to submit
3. open legal consent modal
4. after consent, execute actual submit

If your form framework makes it easier, acceptable alternative:

1. intercept submit
2. open legal modal
3. after consent, run final submit path

In either case:

* user must not be able to submit without consent
* avoid duplicate submissions

DUPLICATE SUBMISSION PROTECTION:

* show loading state on primary button if needed

* form type / form identifier
* legal text version identifier

When modal is approved, save the exact legal version displayed during that approval.

Save the exact version.
